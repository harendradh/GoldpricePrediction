What is a skill

A folder containing a SKILL.md with YAML frontmatter (name, description) plus instructions, templates, and helper scripts. The agent loads it only when the description matches the user's request.

Frontmatter that triggers loading

  • name · unique identifier
  • description · the trigger sentence — keep it specific
  • license · optional

Resolution order

  • Project skills override personal skills with the same name
  • Commit project skills to share with your team
  • Keep personal skills for individual style preferences

Why this gives determinism

Same trigger → same SKILL.md → same procedure → same shape of output. Templates and schemas inside the skill folder act as a hard contract.

What are instructions

Plain markdown rules layered on top of the system prompt. They constrain behavior for a task or a path glob — narrower than the system prompt, broader than a skill.

Scoping patterns

  • Global · applies to every turn in the project
  • Path-scoped · e.g. *.py or spark/**
  • Task-scoped · loaded only for review, test, or doc tasks

Typical contents

  • Coding conventions (no UDFs without justification, explicit schemas)
  • Output format (always emit a markdown table for reviews)
  • Forbidden patterns (no .collect() on large DataFrames)
  • Domain glossary (your team's terms for pipelines, jobs, runs)

Loading order

System prompt → personal instructions → project instructions → path-scoped instructions → user message. Later wins on conflict.

What is a tool

A function the agent can call with structured arguments to do something in the real world — query Databricks, post to Confluence, open a PR, run code.

How tools are registered

  • MCP servers · external processes exposing tools over the Model Context Protocol
  • Built-in · file operations, web search, code execution
  • Custom · your own functions registered in the SDK

Tool config files hold

  • Connection details (URL, auth method)
  • Tool schema (name, parameters, return type)
  • Allowlist/denylist of which tools are exposed
  • Rate limits and side-effect classification

Security note

Project .mcp.json requires explicit approval before tools execute. Personal configs run with your standing permissions.

What is a spec

A machine-verifiable contract for the output shape. The agent's self-check step validates against it before responding — and retries if validation fails.

What lives in a spec

  • Required fields and types
  • Value constraints (patterns, enums, ranges)
  • Cross-field rules (if X then Y is required)
  • Worked examples that also serve as fixtures

Wiring specs into skills

Reference the schema from a skill's reference/ folder so The agent validates output against the same file your CI uses. Single source of truth.

Why this matters for you

For your JSON pipeline profiles, the spec is what makes the agent's output drop-in usable in your existing pipeline. No spec → variable shape → manual cleanup every time.

Prompts vs instructions vs skills

  • Instructions · always-on rules
  • Prompts · reusable, parameterized requests
  • Skills · procedures the agent loads itself when triggered

Best uses for prompts

  • Slash commands in Copilot (/review, /test)
  • Repeated workflows with small variations
  • Onboarding template for new pipelines

Parameterization

Use {{var}} placeholders and pass values at invocation. Prompts can reference skills (@skill:pyspark-job-scaffold) so a single command pulls in the full procedure.

What discovery produces

An in-memory registry: a map of skill name → folder path, instruction file → scope, tool name → endpoint, schema → URI.

Lazy parsing

Only frontmatter is parsed at discovery time. The body of each SKILL.md is read on demand when a trigger matches.

Performance note

Walk is shallow — typically two levels deep. Skills with deeply nested folders won't be auto-discovered.

+
▣ Conflict resolution rules

Same skill name in both scopes:
  project wins → personal silently shadowed

Same instruction key:
  later in load order wins
  (path-scoped > global)

Same tool name:
  project .mcp.json > personal
  explicit allowlist > implicit


▣ Frontmatter parse

---
name: pyspark-code-review
description: Review PySpark
  code against checklist...
version: 2.1.0
---

Why project overrides personal

Teams need consistent output across teammates. Personal preferences are useful defaults but must not break shared workflows.

What gets registered

  • Skill descriptions → trigger index (semantic match later)
  • Instructions → context injection rules
  • Tool schemas → callable handlers
  • Specs → validator instances

Versioning

If a skill exports a version field, the agent prefers the newest semver-compatible version when multiple candidates match.

Step 03 · Assemble context window
budget tokens · prioritize signals · inject grounding
+
▣ Context window layout

┌──────────────────────────────────┐
│  SYSTEM PROMPT                  │
│  · identity, policies            │
├──────────────────────────────────┤
│  RESOLVED INSTRUCTIONS          │
│  · global → path-scoped          │
├──────────────────────────────────┤
│  TRIGGERED SKILLS               │
│  · SKILL.md bodies + refs        │
├──────────────────────────────────┤
│  RAG RETRIEVED CHUNKS           │
│  · top-k from vector + BM25      │
├──────────────────────────────────┤
│  CONVERSATION HISTORY           │
│  · recent N turns, summarized    │
├──────────────────────────────────┤
│  CURRENT USER MESSAGE           │
└──────────────────────────────────┘
       ↓ to reasoning core

Token budget allocation

  • System + instructions · ~5–10%
  • Skills · loaded only when triggered
  • RAG chunks · ~20–40%
  • History · ~10–20%, summarized when long
  • Headroom for response · ~25%

What gets dropped first under pressure

Older conversation turns → unused RAG chunks → less-relevant skill examples. Specs and guardrails are never dropped.

Why deterministic assembly matters

Same inputs → same context layout → same response. The assembler is your most powerful determinism lever after specs.

Step 04 · Plan · act · observe · refine
reasoning loop until self-check passes
+
▣ The inner loop

while not done:
   plan = decompose(request, skills)
   for step in plan:
       result = call_tool(step)
       observe(result)
       if result.error:
           replan()
   draft = synthesize(observations)
   ok = validate(draft, specs)
   if ok and guardrails_pass(draft):
       done = True
   else:
       feedback = critique(draft)
       continue
return draft


▣ Stop conditions

✓ specs validate
✓ guardrails pass
✓ max iterations not hit

Plan

Break the request into ordered steps. Each step maps to a skill or tool call.

Act

Execute the next step. Tool calls run with structured arguments; results stream back.

Observe

Examine the result. Was it an error? Empty? Did it change the plan?

Refine

Update the plan, re-rank remaining steps, or trigger a different skill if the first one didn't fit.

Self-check before emit

  • JSON validates against the spec
  • No prohibited content
  • Citations present where required
  • Confidence above threshold
